# 카메라 상대 렌더링 (Camera-Relative Rendering) 설계

## 배경

리얼 스케일에서 바깥쪽 왜소행성(Pluto, Makemake, Eris)을 더블클릭해 확대하면 두 가지 결함이 보인다:

1. 천체가 심하게 떨린다 (측정값: 프레임당 30.51px)
2. 구가 매끄럽지 않고 각져 보인다 (측정값: 구 표면의 반지름 편차 2~4%)

비주얼 스케일에서는 둘 다 나타나지 않는다 (반지름 편차 0.01%).

## 근본 원인

float32의 정밀도(ULP)는 원점에서 멀어질수록 비례해 나빠지는데, 천체의 반지름은 그대로다.

Eris는 원일점에서 월드 원점(태양)으로부터 **45,985 유닛** 떨어져 있고, 그 지점의 float32 ULP는 **0.0039**다. Eris의 반지름은 **0.09**. 즉 반지름 하나를 표현할 눈금이 23칸뿐이다.

정점 셰이더가 `instanceModel * vec4(inPosition, 1.0)`으로 정점을 월드 좌표(≈45985)에 놓는 순간, 구의 표면이 이 0.0039 격자에 스냅된다 → **각진 구**. 천체가 공전하며 매 프레임 격자에 다시 스냅되므로 → **떨림**.

측정 결과 (실제 `Camera::smoothFollow` 누적기를 구동해 얻음):

| 모드 | faceting (반지름 편차) | jitter |
|---|---|---|
| 현재 (절대 월드 좌표) | 2.611 % | 30.51 px |
| 카메라 상대, float32 CPU | 0.000 % | 26.24 px |
| 카메라 상대, double CPU | 0.000 % | **0.00 px** |

**float32만으로는 불충분하다.** 각진 구는 사라지지만 떨림은 남는다. `camera.target` 자체가 45985 크기의 float32 누적값이라 동일한 0.0039 격자에 갇혀 있기 때문이다.

## 핵심 아이디어

모든 좌표의 기준을 **월드 원점(태양)** 에서 **카메라 타겟**으로 옮긴다. 모델 행렬의 이동값을 `천체위치 − 카메라타겟`으로 주고, 뷰 행렬은 카메라를 원점에 둔다. 그러면 GPU는 `0.5 + 0.09×정점` 같은 작은 수만 다루게 되어 정밀도가 남아돈다.

**전부 double로 바꾸는 것이 아니다.** double은 `천체위치 − 카메라타겟` 이 뺄셈 하나를 위해서만 필요하다. 그 결과(작은 수)는 float로 내려도 정밀도가 충분하다. 카메라에서 먼 물체는 애초에 그만한 정밀도가 필요 없으므로 그대로 둔다.

## 변경 대상

| 구성요소 | 변경 |
|---|---|
| 공전 수학, `Planet::currentPosition`, 태양/달 위치 | `glm::dvec3` |
| `Camera::target`, `Camera::lastNewTarget` | `glm::dvec3` (`pos`는 작은 값이므로 float 유지) |
| 모든 모델 행렬 | 이동값 = `glm::vec3(위치D − 카메라타겟D)` |
| 뷰 행렬 | `glm::lookAt(camera.pos, vec3(0), WORLD_UP)` — 카메라가 원점 |
| UBO `cameraPos` / `sunPos` / `earthPos` / `moonPos` | 카메라 상대 좌표 |
| 레이캐스트(`handleRaycast`), 네임태그(`worldToScreen`) | 동일 상대 좌표계 |
| 소행성 `push.model` | 앞에 `T(−카메라타겟)` 추가 |
| 그림자 `lightSpaceMatrix` (lightView/lightProj) | 상대 좌표계로 구성 |

## 반드시 함께 고쳐야 하는 3곳

좌표계를 바꾸면 **"월드 원점 = 태양"을 암묵적으로 가정한 코드가 조용히 깨진다.** 전수 조사 결과 정확히 세 군데다:

- `shaders/shader.frag:275` 토성 고리: `normalize(-fragPos)` → `normalize(ubo.sunPos - fragPos)`
- `shaders/shader.frag:301` 소행성대/카이퍼벨트 구분: `length(fragPos)` → `length(fragPos - ubo.sunPos)`
- `src/main.cpp` 그림자 초점: `if (glm::length(shadowFocusPos) < 0.1f)` → `if (lockedTargetType == 0)`
  (원점과의 거리로 "태양에 고정된 상태인가"를 판정하고 있다)

세 곳 모두 현재 좌표계에서는 등가이므로 **먼저 명시적으로 바꿔 두면 무해한 사전 작업**이 되고, 이후 좌표계 전환이 안전해진다.

나머지 조명 코드는 전부 `ubo.sunPos - fragPos` 같은 **두 좌표의 차이**로만 쓰고 있어 좌표계를 통째로 옮겨도 그대로 동작한다. `dFdx(fragPos)`/`dFdy(fragPos)`(소행성 플랫 셰이딩)도 미분이라 평행이동에 불변이다. 태양 홍염(type 7)과 토성 고리(type 5)가 쓰는 `vec3(push.model[3])`도 모델 행렬에서 뽑아내므로 좌표계를 따라 자동으로 함께 움직인다.

## 프레임 내 순서 제약

현재 `updateUniformBuffer`는 천체의 **위치 계산과 모델 행렬 생성을 같은 루프에서** 하고, 그 루프는 카메라 업데이트보다 **먼저** 돈다.

모델 행렬이 카메라 상대 좌표를 쓰게 되면 이 순서로는 깨진다. 모델 행렬은 직전 프레임의 `camera.target`을, 뷰 행렬은 이번 프레임의 것을 쓰게 되어 한 프레임 분량의 카메라 이동(리얼 스케일 Eris 기준 5.9 유닛)만큼 어긋나기 때문이다.

따라서 다음 순서로 재구성한다:

1. 모든 천체의 **위치만** double로 계산
2. 카메라 업데이트 (위치를 입력으로 사용)
3. 확정된 `camera.target`으로 **모델 행렬 생성**

## 궤도선

궤도선은 별도 처리가 필요하다. 모델 행렬만 상대 좌표로 바꿔도 GPU가 `34000×cos(θ) − 45985`를 float32로 계산하는 구조라 상쇄 오차(≈14px 떨림)가 그대로 남는다. 이는 반지름 34000짜리 원 위의 점을, 그 원 위에 있는 카메라 기준으로 표현하려는 계산 자체의 성질이다.

**해법:** 현재 고정된(locked) 천체의 궤도선만 매 프레임 CPU에서 double로, 카메라 상대 좌표로 정점을 생성해 전용 동적 정점 버퍼에 올린다.

- 비용: 4096정점 × 44바이트 = 180KB/프레임. 무시할 수준.
- 나머지 행성의 궤도선은 카메라에서 수만 유닛 떨어져 있어 오차가 각도상 보이지 않으므로 기존 공유 지오메트리 + 모델 행렬 방식을 유지한다.
- 정밀도가 필요한 것은 항상 카메라 근처의 궤도선뿐이고, 궤도 반경이 고정이므로 카메라에 가장 가까운 궤도는 정의상 고정된 천체의 궤도다.
- 달의 궤도선은 지구 기준(≈500 유닛)에서 상쇄가 일어나 ULP가 3e-5, 달 반지름 0.13 대비 0.02%라 문제되지 않는다.

버퍼는 기존 `uniformBufferMapped` / `indirectDrawMapped`와 동일한 패턴(생성 시 영구 매핑, 프레임 시작 시 `memcpy`)을 따른다. 앱이 `inFlightFence`로 완전히 직렬화되어 있어(매 프레임 시작 시 fence 대기) GPU가 유휴 상태일 때 쓰므로 안전하다.

## 검증

- **정밀도 회귀:** 이미 작성된 측정 하네스로 확인. 목표 = faceting 0.000%, jitter < 1px.
- **궤도선:** 고정 천체 기준 궤도선 정점의 프레임간 오차 측정.
- **좌표계 변경의 파급:** 조명, 그림자, 클릭 판정(레이캐스트), 네임태그 위치가 좌표계 변경 후에도 정상 동작하는지 — **이 작업에서 가장 위험한 부분**이다. 태스크마다 빌드 + 스크린샷 육안 확인을 건다.
- 이 저장소에는 자동화된 테스트 스위트가 없다. "테스트"는 (a) 빌드 성공, (b) 실행 후 스크린샷 육안 확인, (c) 별도 측정 하네스 실행을 의미한다.
- **알려진 제약:** 현재 자동화 세션에서는 앱에 마우스 클릭 주입이 되지 않는다(여러 Windows 입력 API 전부 실패, 환경 제약으로 확인됨). 따라서 "왜소행성을 더블클릭해 확대"하는 최종 확인은 사용자가 직접 수행해야 한다. 스크린샷 캡처는 `PrintWindow`(PW_RENDERFULLCONTENT)로 가능하므로 정지 화면 확인은 자동화 가능하다.

## 범위 밖 (명시적으로 하지 않는 것)

- 소행성대의 정밀도: 소행성은 리얼 스케일에서 `beltScale`(200배)로 함께 확대되므로 크기 1.6~50 유닛, 거리 최대 20000 → ULP 0.002 대비 0.12%로 영향 없다. 좌표계만 맞춰주고 정밀도 작업은 하지 않는다.
- 고정되지 않은(locked가 아닌) 천체들의 궤도선 정밀도: 위 "궤도선" 항목 참조.
- 리얼 스케일의 거리/크기 수치 자체는 변경하지 않는다.
