# 별자리(Constellation) 구현 설계

## 배경

Ephemeris의 하늘 배경은 큐브맵 텍스처 두 장이다 — `skybox_stars.dds`(밝은 별)와
`skybox_band.dds`(은하수 + 어두운 별). NASA/Gaia 실측을 기반으로 실제 별이 실제
위치에 **그려진 이미지**지만, 별 하나하나의 좌표·등급·이름을 담은 **카탈로그는
없다.** 셰이더는 방향으로 픽셀을 읽을 뿐(`texture(skyboxTex, fragTexCube)`), "이
픽셀이 베텔게우스"라는 정보를 갖고 있지 않다.

따라서 별자리 선을 그으려면 데이터를 **추가**해야 한다: 밝은 별의 좌표 카탈로그와,
어느 별을 어느 별과 잇는지의 선 정의.

## 목표

배경 하늘 위에, 실제 별에 정확히 얹히는 **IAU 88개 별자리**의 선과 라틴명 이름을
그린다. 켜고 끌 수 있고, 마우스로 별을 짚으면 그 별에서 별자리가 자라나듯 펼쳐진다.

## 핵심 발견 — 하늘은 회전한다

`recordSky`는 배경에 `skyMat = rotate(currentAppTime * 0.5°, Y축)`을 적용한다. 배경
하늘이 앱 시간에 따라 초당 0.5도씩 천천히 돈다. **별자리 선도 같은 `skyMat`를
적용해야** 배경 별에 계속 붙어 함께 돈다. 이 하나로 정렬 유지 문제가 풀린다.

## 사용자 결정 사항

| 항목 | 결정 |
|---|---|
| 표시 내용 | 선 + 라틴명 이름. 데이터 구조는 나중에 그림(figure art)을 담을 수 있게 확장 가능하게 설계 |
| 이름 언어 | 영문/라틴명 (현재 ImGui 폰트로 바로 됨. 한글 폰트 불필요) |
| 범위 | IAU 88개 전부 |
| 표시 방식 | 설정 토글(전체) + 마우스 호버(개별 강조) — 둘 다 |
| 렌더 방식 | A안: 선만 얹기(배경 별에 연결). 우리 별점은 그리지 않음 |
| 호버 애니메이션 | 호버한 별을 뿌리로 BFS 물결처럼 선이 자라남 |
| 호버 해제 | 페이드 아웃 |
| 선 색 | 은은한 하양+보랏빛(라벤더). 호버 강조 시 더 밝고 보랏빛으로 |

## 데이터

### 밝은 별 카탈로그

별자리 선이 잇는 별들의 **적경(RA)·적위(Dec)·등급·ID**. 88개 선이 참조하는 별은
수백 개 수준이라 파일이 작다. Hipparcos/HYG 계열의 자유 라이선스 데이터에서 선에
쓰이는 별만 추린다.

### 별자리 선 정의

별 쌍(간선)의 목록. Stellarium `constellationship`(서양/현대) 세트가 88개 전부를
표준 연결로 담고 있다. **라이선스를 확인해 허용되는 세트를 고르고 README 크레딧에
출처·라이선스를 표기한다.** (이 프로젝트는 출처를 엄격히 표기한다 —
[[accuracy-over-appearance]]와 같은 원칙.)

### 저장 위치

`data/constellations/`에 작은 텍스트 파일로 두고 저장소에 커밋한다. 텍스처와 달리
작으므로 Release 배포가 아니라 저장소에 직접 넣어 클론하면 바로 쓰인다.

### 정렬은 측정으로 정한다 (짐작 금지)

배경 큐브맵이 방향을 어떤 좌표계로 매핑하는지는 큐브맵을 구운 스크립트의 규약
(적도좌표/은하좌표, 경도 방향, 축 배치)에 달렸는데, 그 규약이 현재 코드에 남아 있지
않다. 따라서:

1. 시리우스·베텔게우스·리겔 등 **밝고 명확한 별 몇 개**를 후보 규약으로 방향 계산.
2. 그 방향의 큐브맵 픽셀이 실제로 밝은지 확인.
3. 맞는 규약을 찾아 **RA/Dec → 방향 벡터 변환 함수**를 확정한다.

이 측정이 통과해야 나머지가 의미가 있으므로 **구현 계획의 첫 검증 단계**로 둔다.
선이 실제 밝은 점에서 어긋나면 실패로 본다.

## 아키텍처

### 새 모듈: `src/sim/StarCatalog.{hpp,cpp}` (순수 데이터, Vulkan 무관)

책임: 카탈로그와 별자리 그래프를 로드하고 RA/Dec를 방향 벡터로 변환한다.

```cpp
struct CatalogStar {
    int   id;            // Hipparcos 번호 등 원본 ID
    glm::vec3 dir;       // 큐브맵 프레임의 단위 방향 벡터 (검증된 규약으로 변환)
    float magnitude;
};

struct Constellation {
    std::string name;                 // 라틴명 (예: "Orion")
    std::vector<glm::ivec2> edges;    // CatalogStar 인덱스 쌍
    std::vector<std::vector<int>> adjacency;  // 성장 애니메이션용 인접 리스트
    glm::vec3 centroidDir;            // 이름 라벨을 놓을 중심 방향
    // 확장 여지: 나중에 figure art(폴리곤/텍스처)를 여기에 추가한다.
};

class StarCatalog {
public:
    bool load(const std::string &starPath, const std::string &linePath);
    const std::vector<CatalogStar>  &stars() const;
    const std::vector<Constellation> &constellations() const;
    // RA(시간각 또는 도)·Dec(도) → 큐브맵 프레임 단위 방향. 정렬 검증으로 확정.
    static glm::vec3 raDecToDir(double raDeg, double decDeg);
    // 마우스 광선 방향에 가장 가까운 별과 그 별자리를 각도로 찾는다.
    // 반환: 별자리 인덱스와 그 안의 별 인덱스(뿌리). 없으면 -1.
    int pickNearest(const glm::vec3 &rayDir, float maxAngleRad,
                    int &outStarIndex) const;
};
```

`raDecToDir`는 이 프로젝트에서 드물게 단위 테스트가 가능한 순수 함수다. 알려진 별
방향으로 수치 확인한다.

### 렌더링 통합 (기존 파일)

**선** — 기존 `linePipeline`을 재활용한다. 궤도선(objectType 6)과 구분되는 **새
objectType 하나**(예: 11)를 셰이더에 추가하고, `customData`를 선의 **밝기/알파**로
쓴다. 선 정점의 `color` 속성에 라벤더 기본색을 싣는다.

- 별자리 별 방향에 배경 구(먼 반지름) 위 정점을 놓고, **각 간선을 큰 원(great
  circle)으로 몇 점 분할**해 천구 표면을 따라 휘게 한다(직선 현은 천구를 파고든다).
- 배경과 같은 `skyMat` 회전을 모델 행렬로 넘겨 함께 돈다.
- 하늘 패스에서 배경 깊이로 그려, 앞을 지나는 행성이 자연스럽게 가린다.

**두 개의 정점 버퍼:**
- 정적 버퍼 — 토글이 켜졌을 때의 88개 전체 선(완성된 형태). 로드 시 한 번 만든다.
- 동적 버퍼 — 호버로 **성장 중인** 별자리 하나. 매 프레임 "지금까지 그려진 분할점
  까지"만 다시 채운다.

**이름 라벨** — 기존 `worldToScreen`로 별자리 중심 방향을 화면에 투영하고, 화면
안이며 카메라 앞쪽일 때만 ImGui로 라틴명을 그린다(정보 패널과 같은 방식).

## 상호작용

### 토글

설정창에 `Constellations` 체크박스. `Real Stars`처럼 **독립 축**이라 축척·Real Stars와
무관하게 동작한다. 뷰 토글 관례대로 **세션 전용, 기본 꺼짐, 저장 안 함**. 켜고 끌 때
짧은 페이드(`constLerp`, `starsLerp`와 같은 방식)로 부드럽게.

- 토글 켜짐 → 88개가 은은하게 다 보인다.
- 토글 꺼짐 → 안 보이다가, 호버 시 그 별자리 하나만 떠오른다(탐색 모드).

### 호버 판정

매 프레임 마우스 광선 방향을 구하고 `StarCatalog::pickNearest`로 각도상 가장 가까운
별과 그 별자리를 찾는다. 배경 별은 무한히 먼 거리라 구 교차가 아니라 **방향 내적**만
하면 된다 — 수백 개라 부담 없다. 앞에 행성이 있으면(기존 피킹) 행성 선택이 우선한다.

### 성장 애니메이션 (BFS 물결)

별자리를 그래프로 본다(별 = 꼭짓점, 선 = 간선).

- 호버한 별을 **뿌리**로 BFS를 돌려 각 간선에 **그래프 거리 기반 시작 지연**을 준다.
- 각 간선은 시작 지연 뒤 0.2~0.3초에 걸쳐 자란다(뿌리 쪽 끝에서 이웃 별 쪽으로 채워짐).
- 그래서 가까운 선부터 순서대로 펼쳐지고, 물결이 바깥으로 번진다.
- 매 프레임 동적 버퍼에 "각 간선의 현재 진행도까지의 분할점"만 내보내면 자연스럽게
  자라는 선이 된다. 성장 중인 별자리는 한 번에 하나뿐이라 비용이 작다.

### 호버 해제

페이드 아웃(성장한 상태에서 알파가 0으로). 되감기는 하지 않는다.

### 색

- 기본 선: 은은한 하양+보랏빛(라벤더, 대략 `vec3(0.82, 0.80, 0.95)` 근처).
- 호버 강조: 같은 색을 더 밝고 보랏빛 쪽으로(customData 밝기 상향). 켜둔 나머지는
  더 은은하게.
- 정확한 색조·밝기는 화면에서 보며 정하는 미관 값. 측정 대상이 아니므로 사용자가
  육안으로 확정한다.

### 배경 밝기

건드리지 않는다. 선 자체의 밝기·굵기로 별밭 위에서 읽히게 한다(배경을 어둡게 깔면
더 잘 보이지만 사실성을 해친다 — 필요하면 나중에 옵션).

## 검증

테스트 프레임워크가 없으므로 검증은 **빌드 + 육안**이 기본이나, 두 가지는 측정으로
자동화한다.

1. **정렬 검증(선행 조건).** 밝은 별 몇 개의 방향에서 큐브맵 픽셀이 밝은지 스크립트로
   확인해 `raDecToDir` 규약을 확정한다. 통과 전에는 나머지 작업이 의미 없다.
2. **좌표 변환 수치 확인.** `raDecToDir`를 알려진 별 방향으로 확인.
3. **육안** — 토글 시 88개 선·이름이 실제 밝은 별에 얹히는지, 하늘 회전을 따라가는지,
   호버 시 짚은 별에서 별자리가 자라나는지, 뗄 때 페이드 아웃되는지, 색이 은은한
   라벤더인지.

## 하지 않는 것 (YAGNI)

- **별자리 그림(figure art)** — 이번엔 선+이름만. 단 데이터 구조는 나중에 그림을
  담을 수 있게 열어 둔다(`Constellation`에 확장 여지).
- **우리 별점 직접 렌더(B안)** — 배경 별과 겹쳐 두 개로 보일 위험이 크고 일거리가
  많다. 배경 별에 선만 얹는다.
- **한글 이름/폰트** — 라틴명만. ImGui 한글 폰트 도입은 별개 작업.
- **배경 디밍 옵션** — 기본은 안 한다.
- **호버 되감기 애니메이션** — 페이드 아웃으로 충분.
